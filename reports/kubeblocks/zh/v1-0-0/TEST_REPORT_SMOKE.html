测试方法

不同的运维操作有不同的执行方法，大部分运维操作可以通过提交 OpsRequest 实现，操作详情可参考 https://github.com/apecloud/kubeblocks/tree/main/examples。

创建集群

工作原理：

KubeBlocks 通过 Kubernetes Operator 模式部署数据库引擎组件，并基于自定义资源（如 ComponentDefinition 和 ComponentVersion）创建数据库集群。控制器会根据配置自动调度并维护数据库实例的状态。

实际意义：

  • 验证数据库集群能否成功初始化并进入正常运行状态。
  • 确保数据库服务在启动后可被访问且数据持久化机制有效。
  • 为后续运维操作提供基础验证。

测试用例

```yaml
apiVersion: apps.kubeblocks.io/v1
kind: Cluster
metadata:
  name: example-mysql-cluster
  namespace: demo
spec:
  clusterDef: mysql
  topology: semisync
  terminationPolicy: Delete
  componentSpecs:
    - name: mysql
      serviceVersion: 8.0.35
      replicas: 2
      resources:
        limits:
          cpu: '0.5'
          memory: 0.5Gi
        requests:
          cpu: '0.5'
          memory: 0.5Gi
      volumeClaimTemplates:
        - name: data
          spec:
            storageClassName: ""
            accessModes:
              - ReadWriteOnce
            resources:
              requests:
                storage: 20Gi
```

重启集群

工作原理：

通过修改集群资源对象的状态字段来控制数据库实例的重启行为。KubeBlocks 支持重启集群中的所有 Pod，重启时会先逐个删除再重新创建数据库 Pod。

当数据库出现异常时，也可以尝试重启集群。

实际意义：

  • 验证数据库重启流程是否安全可靠。
  • 确保重启过程中不会导致数据损坏或丢失。

测试用例

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-restart-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: Restart
  restart:
  - componentName: mysql
```

停止/启动

工作原理：

通过修改集群资源对象的状态字段来控制数据库实例的停止/启动行为。停止时会优雅关闭所有数据库 Pod。

停止集群后，其计算资源将被释放，也就是说 Kubernetes 的 Pod 将被释放，但其存储资源仍将被保留。您也可以重新启动该集群，使其恢复到停止集群前的状态。

实际意义：

  • 验证数据库停止/启动流程是否安全可靠。
  • 确保停止/启动过程中不会导致数据损坏或丢失。
  • 提供对非活跃集群进行按需停止的能力，节省资源消耗。

测试用例

停止

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-stop-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: Stop
```

启动

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-start-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: Start
```

垂直扩缩容

工作原理：

通过更改资源需求和限制（CPU 和 Memory）来垂直扩展集群，KubeBlocks 协调底层 InstanceSets 控制器完成垂直扩缩容。

  • 原地更新: 集群不会重启，降低实例更新时对系统可用性的影响。
  • 非原地更新: KubeBlocks 控制器会检测变更并触发滚动更新或热更新策略，确保变配生效同时不影响服务可用性。

实际意义：

  • 验证系统是否能够动态调整 CPU/Memory 资源以应对负载变化。
  • 确保垂直扩缩容时不会导致数据丢失或服务中断。

测试用例

垂直扩容

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-vscale-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: VerticalScaling
  verticalScaling:
  - componentName: mysql
    requests:
      cpu: '1'
      memory: 1Gi
    limits:
      cpu: '1'
      memory: 1Gi
```

垂直缩容

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-vscale-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: VerticalScaling
  verticalScaling:
  - componentName: mysql
    requests:
      cpu: '0.5'
      memory: 0.5Gi
    limits:
      cpu: '0.5'
      memory: 0.5Gi
```

水平扩缩容

工作原理：

KubeBlocks 支持水平扩容和水平缩容两类操作，同时支持副本（replica）和具体实例（instance）的扩缩容。

  • 水平扩容: 支持水平扩容指定副本及指定实例重新上线。
  • 水平缩容：支持水平缩容指定副本及指定实例下线。

可通过声明式 API 或创建 OpsRequest 的方式执行水平扩缩容：

  • 通过声明式 API 方式，可直接修改 Cluster YAML 文件来指定每个组件的副本数量和实例模板。如果新的副本数量大于当前 Pod 的数量，则表示扩展（scale-out）；相反，如果新的副本数量少于当前 Pod 的数量，则表示缩容（scale-in）。
  • 另一种方法是在 OpsRequest 中指定副本数量的增量。控制器（controller）将根据集群组件中当前 Pod 的数量和增量值计算所需的副本数量，并相应地执行扩缩容操作。

实际意义：

  • 验证系统是否能够动态调整资源以应对负载变化。
  • 确保水平扩容后新节点能正确加入集群并分担负载。
  • 确保缩容时不会导致数据丢失或服务中断。

测试用例

水平扩容

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-scale-out-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: HorizontalScaling
  horizontalScaling:
  - componentName: mysql
    scaleOut:
      replicaChanges: 1
```

水平缩容

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-scale-in-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: HorizontalScaling
  horizontalScaling:
  - componentName: mysql
    scaleIn:
      replicaChanges: 1
```

磁盘扩容

工作原理：

通过修改 PVC 请求的存储大小，触发底层存储系统的扩容操作。KubeBlocks 确保数据库文件系统及数据目录同步扩展。

实际意义：

  • 验证存储层弹性扩展能力。
  • 确保在数据增长时无需停机即可扩展存储容量。
  • 提升数据库集群的长期可用性和稳定性。

测试用例

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-expand-volume-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: VolumeExpansion
  volumeExpansion:
  - componentName: mysql
    volumeClaimTemplates:
    - name: data
      storage: 30Gi
```

实例重建

工作原理：

针对某个异常数据库实例执行重建操作，KubeBlocks 删除目标 Pod，然后基于原有数据重建数据库实例。

  • 原地重建: 使用相同的 pod 名称重建实例。
  • 非原地重建: 将通过水平扩容来创建新实例，并在新实例准备就绪后删除该实例。

实际意义：

  • 验证单个数据库实例的重建能力。
  • 确保重建后的实例能快速恢复并重新加入集群。
  • 提高数据库系统的自我修复能力。

测试用例

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-rebuildinstance-ops
  namespace: demo
spec:
  type: RebuildInstance
  clusterName: example-mysql-cluster
  force: true
  rebuildFrom:
    - componentName: mysql
      instances:
        - name: example-mysql-cluster-mysql-0
      backupName: backup-demo-example-mysql-cluster-20250528130909
      inPlace: false
      restoreEnv:
        - name: BACKUP_FOR_STANDBY
          value: "true"
```

主备切换

工作原理：

数据库主备切换是指在数据库集群中将主数据库的角色切换到备用数据库的过程，使备用数据库成为新的主数据库实例。

通常在主数据库故障、维护或升级时执行主备切换操作，以确保数据库服务的高可用性和连续性。

实际意义：

  • 验证主备切换流程是否可控。
  • 确保切换过程中数据一致性不受影响。
  • 支持计划内维护、负载均衡等场景。

测试用例

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-switchover-ops
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  type: Switchover
  switchover:
  - componentName: mysql
    instanceName: example-mysql-cluster-mysql-0
```

备份/恢复

工作原理：

KubeBlocks 提供备份恢复功能，以确保数据的安全性和可靠性。KubeBlocks 的备份恢复功能依赖于 BackupRepo，在使用完整的备份恢复功能之前，首先需要配置 BackupRepo。

KubeBlocks 采用物理备份的方式，将数据库中的物理文件作为备份对象。你可以根据实际需求，选择对应的方式按需或定时备份集群数据。

  • 按需备份：根据不同的备份选项，按需备份可以进一步分为备份工具备份和快照备份两种。
    • 备份工具备份：可使用数据库产品的备份工具，如 MySQL XtraBackup 和 PostgreSQL pg_basebackup。KubeBlocks 支持为不同的数据产品配置备份工具。
    • 快照备份：如果你的数据存储在支持快照的云盘中，你可以通过快照创建数据备份。快照备份通常比备份工具备份更快，因此推荐使用。
  • 定时备份：可指定保留时间、备份方法、时间等参数来自定义备份设置。

KubeBlocks 支持从备份集中恢复数据。

实际意义：

  • 验证数据保护机制的有效性。
  • 确保在灾难恢复场景下可以快速还原数据。
  • 符合企业级 SLA 对数据安全性的要求。

测试用例

备份

```yaml
apiVersion: dataprotection.kubeblocks.io/v1alpha1
kind: Backup
metadata:
  name: example-mysql-cluster-backup
  namespace: demo
spec:
  backupMethod: xtrabackup
  backupPolicyName: example-mysql-cluster-mysql-backup-policy
  deletionPolicy: Delete
```

恢复

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-restore
  namespace: demo
spec:
  clusterName: example-mysql-cluster-restored
  force: false
  restore:
    backupName: example-mysql-cluster-backup
    backupNamespace: demo
  type: Restore
```

参数配置

工作原理：

KubeBlocks 支持数据库参数配置动态渲染。

配置动态渲染功能简化了数据库规格调整的过程。用户无需手动更改数据库参数，KubeBlocks 会自动处理参数的更新和配置，以适应新的规格。这样可以节省时间和精力，并减少由于参数设置不正确而导致的性能问题。

需要注意的是，配置动态渲染功能并不适用于所有参数。有些参数可能需要手动进行调整和配置。

此外，如果你对数据库参数进行了手动修改，KubeBlocks 在更新数据库参数模板时可能会覆盖手动修改。因此，在使用动态调整功能时，建议先备份和记录自定义的参数设置，以便在需要时进行恢复。

实际意义：

  • 验证数据库参数动态调整能力。
  • 支持根据不同业务需求灵活优化数据库性能。
  • 提高运维效率，降低人工配置错误风险。

测试用例

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: mysql-reconfigure-dynamic
  namespace: demo
spec:
  clusterName: example-mysql-cluster
  force: false
  reconfigures:
  - componentName: mysql
    parameters:
      - key: max_connections
        value: '100'
  preConditionDeadlineSeconds: 0
  type: Reconfiguring
```

服务暴露

工作原理：

通过创建或删除 Service 资源，控制数据库服务是否对外暴露。支持 LoadBalancer、NodePort 等多种方式。

实际意义：

  • 验证网络服务暴露/隐藏的安全性和即时性。
  • 支持临时开放访问权限用于调试或集成测试。
  • 提高数据库服务的访问控制粒度。

测试用例

开启

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-expose-enable-ops
  namespace: demo
spec:
  type: Expose
  clusterName: example-mysql-cluster
  expose:
  - componentName: mysql
    services:
    - name: internet
      serviceType: LoadBalancer
      annotations:
        service.beta.kubernetes.io/aws-load-balancer-type: nlb
        service.beta.kubernetes.io/aws-load-balancer-internal: "false"  # or "true" for an internal VPC IP
      roleSelector: primary
    switch: Enable
```

关闭

```yaml
apiVersion: operations.kubeblocks.io/v1alpha1
kind: OpsRequest
metadata:
  name: example-mysql-cluster-expose-disable-ops
  namespace: demo
spec:
  type: Expose
  clusterName: example-mysql-cluster
  expose:
  - componentName: mysql
    services:
    - name: internet
      roleSelector: primary
      serviceType: LoadBalancer
    switch: Disable
  preConditionDeadlineSeconds: 0
```

更新集群

工作原理：

更新集群资源配置（如 Monitor 开关、TerminationPolicy 等），KubeBlocks 控制器会检测变更并触发滚动更新或热更新策略，确保配置生效同时不影响服务可用性。

实际意义：

  • 支持动态调整集群参数，提升运维灵活性。
  • 验证配置更新过程中的服务连续性和一致性。
  • 减少手动干预，实现自动化配置管理。

连接数据库

工作原理：

使用标准客户端连接数据库服务，验证集群是否可被正常访问。

实际意义：

  • 验证数据库服务端口可达性和认证机制正常。
  • 确认集群处于健康状态并能接受连接请求。
  • 作为其他运维操作的基础依赖项进行验证。

性能测试

工作原理：

通过内置或外部测试工具对数据库集群进行基准性能测试，评估其吞吐量、响应时间等指标。

实际意义：

  • 验证数据库在典型负载下的性能表现。
  • 用于不同版本或配置间的性能对比。
  • 支持容量规划和性能调优。
